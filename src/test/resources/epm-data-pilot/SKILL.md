# EPM 数据查询与分析（epm-data-pilot）

## Description

本 Skill 用于在 EPM（企业绩效管理）/ ERP 系统中进行预算财务数据的查询、追踪、对比、分析。通过调用后端 MCP 工具链，完成从自然语言提问到结构化财务数据返回的全流程，支持数据溯源、多维度钻取、组织结构贡献分析和预实对比。

核心能力包括：
- **数据查询**：将用户自然语言问题转换为维度成员匹配，获取指定模板的数据。
- **数据追踪**：对聚合（AGG）或公式（RULE）类型的单元格向上追溯，展示完整的依赖链路。
- **数据分析**：按组织、科目、期间等维度逐层拆解，辅助用户理解数据构成。
- **预实对比**：利用 `otherCombinationList` 对比实际数与多个预算版本。

## When to Use

- 用户询问预算、财务、EPM、ERP 相关数据，例如："广州公司 1-4 月营业利润是多少"
- 用户要求追踪数据来源或构成，例如："营业利润是怎么算出来的"
- 用户需要按组织/科目/期间维度拆解数据，例如："按分公司看利润贡献"
- 用户提到预实对比、预算 vs 实际，例如："实际数和年度预算对比"
- 用户要求对比多期间、多组织的数据，例如："对比 1-3 月和 4-6 月的数据"
- 用户输入包含以下关键词：预算、财务、EPM、ERP、查询、追踪、分析、对比、预实、forecast、budget、actual

## When NOT to Use

- 用户问题与预算财务数据无关（如人事、生产排程、供应链），应路由给对应的专用 Skill
- 用户仅要求生成报表文件（Word/Excel/PPT），应使用 `docx` / `xlsx` / `pptx` Skill
- 用户要求审查 Skill 定义本身的质量，应使用 `skill-review` Skill
- 用户要求对代码进行审查，应使用 `code-review` Skill
- 用户未提供任何与财务数据相关的上下文且意图不明确时，应先询问用户具体需求，而非直接调用数据工具

## Workflow

> **工具调用参数规范、返回结构和期间编码规则详见 `reference/tool-reference.md`。**

### 场景01：创建会话 id，明确体系
当上下文中找不到 `sessionId` 或者体系编码 `modelNumber` 时：
- step1: **并行调用** `bgm_ai_globalId_create` 和 `bgm_ai_model_query`，同时获取 sessionId 和体系列表
- step2: 当体系数量大于 1 时，必须要求用户选择体系名称，明确体系编码 `modelNumber`；若仅有 1 个体系则直接使用

**进度反馈：**
- step1 完成后 → 向用户汇报"已连接到系统"
- step2 用户确认体系后 → 向用户汇报"已连接到 [体系名称]"

**失败处理：**
- 若 `bgm_ai_model_query` 返回空列表 → 提示"当前账号暂无 EPM 数据权限，请联系管理员开通"，结束本次调用
- 若 `bgm_ai_globalId_create` 调用失败 → 重试一次；再次失败则提示"获取会话id失败，请稍后重试"

### 场景02：数据查询
当用户需要查询新数据（确保要查询的数据在历史会话中不存在）：

- step1: 调用 `bgm_ai_member_match`，传入体系编码 `modelNumber` 和用户提问中的关键字 `userQuestion` → 输出：候选维度成员列表
- step2: 结合step1 返回的候选编码信息和用户提问，匹配用户提问中包含的维度编码和维度成员编码 → 输出：`dimAndMemberList`
- step3: 调用 `bgm_ai_template_query` 传入 `modelNumber`、`dimAndMemberList`（如有） → 输出：候选模板列表
- step4: 结合用户问题与step3 返回的模板，选择一个模板（注意：如果存在多个模板内容或名称相似可要求用户澄清） → 输出：`templateId`
- step5: 调用 `bgm_ai_template_structure_query`，传入 `templateId` 和 `dimAndMemberList`（如有） → 输出：完整维度和维度成员
- step6: 当step5 返回的维度可选成员数量小于4个时可以直接使用“批量查询”模式，可选成员大于4时要求用户澄清 → 输出：最终 `dimAndMemberList`
- step7: 调用 `bgm_ai_data_trace` 工具，传入所有已匹配的维度和维度成员编码信息（确保与step5 返回的维度数量一致） → 输出：单元格数据

**member_match 超时与部分匹配策略：**
- `bgm_ai_member_match` 可能返回 `Read timed out` 错误（超时），此时：
  - 如果同一个会话中已有其他关键词匹配成功并返回了可用维度成员，直接利用已有的匹配结果，不再重试超时的调用
  - 如果所有关键词均超时且无任何匹配结果，跳过 member_match，直接使用 template_query 配合已知的模板 ID 或科目编码查询模板
  - 不需要反复重试超时的 member_match 调用，避免浪费时间和资源
- 当部分维度通过 member_match 匹配成功（如科目维度），但另一部分维度匹配失败（如组织维度超时）时：
  - 在 template_structure_query 返回的维度结构中，通过维度成员名称人工匹配用户提问中的组织名称
  - 例如：用户说"广州公司"，模板结构中存在"广州分公司"(HYJD_GZ)，则可直接使用

**多期间/多组织批量查询优化：**
- 当用户提问涉及多个月份（如"1-4 月"）时，**将多个期间的查询打包到同一次 `bgm_ai_data_trace` 调用的 items 数组中**，而非分多次调用
- 示例：查询 1-4 月数据时，items 中应包含 4 个对象，每个对象的 dimAndMemberList 仅期间不同

```json
{
    "params": {
        "sessionId": "xxx",
        "modelNumber": "YS_DEMO_AI",
        "items": [
            {"dimAndMemberList": [{"dimNumber": "BudgetPeriod", "memberNumber": "FY2026.M01"}, ...]},
            {"dimAndMemberList": [{"dimNumber": "BudgetPeriod", "memberNumber": "FY2026.M02"}, ...]},
            {"dimAndMemberList": [{"dimNumber": "BudgetPeriod", "memberNumber": "FY2026.M03"}, ...]},
            {"dimAndMemberList": [{"dimNumber": "BudgetPeriod", "memberNumber": "FY2026.M04"}, ...]}
        ]
    }
}
```

### 场景03：继续追踪已知单元格
当用户已经查询到部分数据时，要求继续追踪数据时：
**注意：仅当单元格类型为聚合（AGG）或公式（RULE）可以继续追踪，录入（INPUT）类型单元格无法追踪。**
- step1: 首先检查历史会话中是否已经包含用户想要追踪的数据，聚合（cellType:AGG）单元格判断其 `aggregationDim` 中是否已包含目标方向的 children 数据，公式单元格（cellType:RULE）判断其rule对象的children 数据
   - 若已包含：直接使用现有数据，无需再次调用工具
   - 若未包含（children 为空或不存在）：继续下一步
- step2: 获取被追踪单元格的 `id`（来源：`bgm_ai_data_trace` 返回结果）
- step3: 调用 `bgm_ai_data_node_trace` 工具，传入单元格 `id`、会话 `sessionId` → 输出：包含该节点及其上游节点数据的完整信息

**关于 aggregationDim 的 children 数据复用说明：**
- `aggregationDim` 是一个数组，按维度分组（如组织维度、科目维度），每个维度下有对应的 `children` 数组
- 当用户要求"继续追踪"时，注意区分两个方向：
  - **组织维度方向**：在 `aggregationDim` 中找到 `dimName === "组织"`（或 `dimNumber === "Entity"`）的条目，其 `children` 即为下级组织的数据
  - **科目维度方向**：在 `aggregationDim` 中找到 `dimName === "科目"`（或 `dimNumber === "Account"`）的条目，其 `children` 即为下级科目的数据
- 例如：营业利润单元格的 `aggregationDim` 可能同时包含组织维度的子节点（如建发九龙仓）和科目维度的子节点（如营业收入、营业总成本）。根据用户追踪方向选择不同的 `children` 来展示，无需重复调用工具
- 不同维度的 children 中，子节点的 `description` 可用来区分该节点所属的具体维度和成员

**data_node_trace 返回结构解读：**
- `bgm_ai_data_node_trace` 返回的 `fullCellRelationList` 是**以被查询节点为起点的完整上下游树**（包含该节点、其父节点、兄弟节点及子节点的全部数据），而非仅当前节点
- 这意味着：一次 data_node_trace 调用会返回已经包含过的数据（如父节点数据），使用时应从 `cellMatchList` 中找到本次查询的目标节点 `id`，以该节点为起点向子节点方向展开
- 在展示追踪结果时，应聚焦于目标节点及其下游 newly discovered 的数据，避免重复展示用户已经看过的上游链路

**多级聚合维度展开策略（如产品业态）：**
- 当追踪过程中遇到按产品业态聚合的节点时（如"业态合计"→"住宅"→"超高层住宅/中高层住宅/高层住宅"），需要逐层调用 data_node_trace 展开
- 第一层：业态合计节点 → 得到子业态（如"住宅"）
- 第二层：住宅节点 → 得到更细分的业态（如"超高层住宅""中高层住宅""高层住宅"）
- 每一层的 children 节点如果是 RULE（公式）类型，且其 rule.children 指向下级聚合数据，则直接利用 rule.children 的数据展开即可，无需再为 RULE 节点本身调用 data_node_trace（除非需要查看 RULE 的计算公式明细）
- 每层展开后，用表格展示各业态的数值，并可联动展示 otherCombinationList 中的预算数据进行预实对比

**进度反馈：**
- step1 判断已有数据满足需求 → 向用户汇报"已获取到关联数据，正在整理展示"
- step3 调用 data_node_trace → 向用户汇报"正在追踪上游数据，请稍候"
- 数据返回后 → 按对应格式输出并附加文字总结

### 场景04：替换已知单元格的部分维度继续追踪
当用户已经追踪到部分数据，需要替换部分维度继续追踪时：
例如：已经获取了广州公司的营业利润数据，用户提问："深圳分公司的数据呢？"

- step1: 获取被追踪单元格的 `id`（来源：`bgm_ai_data_trace` 返回结果）
- step2: 调用 `bgm_ai_member_match`，传入体系编码 `modelNumber` 和用户提问中的关键字（例如：深圳公司） `userQuestion` → 输出：候选维度成员列表（利用候选维度列表活动深圳公司编码）
- step3: 调用 `bgm_ai_data_node_trace` 工具，传入单元格 `id`、会话 `sessionId`，以及需要调整的维度和编码（这里是深圳公司的维度编码） `dimAndMemberList` → 输出：替换组织维度为深圳公司后的查询数据

### 场景05：数据分析（含组织结构/科目结构追溯）

当用户需要分析某个数据的来源、构成或差异时：

- step0: 判断当前会话中已获取的数据是否满足需求
    - 满足：直接回复用户
    - 不满足：继续step1
- step1: 判断当前会话中是否存在与用户提问关联的源头数据
    - 不存在：参考场景02 获取源头数据
    - 存在：参考场景03 追踪上游数据
- step2: 参考 场景03 追踪持续上游数据，直到获取的数据已经满足需求
- step3: 整理完整的追溯链路输出

### 场景06：预实分析

当用户提到预实分析、对比实际数、对比预算数、预算目标达成率、时调用：
- step0: 判断当前会话中已获取的数据是否满足需求
    - 满足：直接回复用户
    - 不满足：继续step1
- step1: 参考 场景02 获取用户提问关联的源头数据，涉及数据类型和版本均采用实际数，这样可以直接在`otherCombinationList`属性中查到预算数据和版本
- step2: 单元格数据中 `otherCombinationList` 用于存储除当前版本外的实际数和其它版本的预算数据（注意：任何数据都只有一个实际数版本，可能有多个预算数版本）
- step3: 对比当前版本与 `otherCombinationList` 中的数据，计算差异金额和差异率，输出对比表格

**连续差异化分析策略：**
- 当用户先查询了实际数，然后要求"对比预算数"时，优先使用**当前会话中已获取的单元格数据的 `otherCombinationList`**，无需重新查询
- 如果 `otherCombinationList` 中存在多个预算版本（如 ANNUAL_V1、ANNUAL_V2），默认使用第一个版本进行对比，同时向用户提示存在其他版本可供选择
- 差异率计算公式：`差异率 = (实际数 - 预算数) / 预算数 × 100%`，保留 2 位小数
- 差异额正负含义：正数表示实际优于预算（节约/超收），负数表示实际劣于预算（超支/短收）
- 输出对比表格时，在表格下方附加简要文字总结，指出差异的主要驱动因素（如"营业总成本超支是导致营业利润未达预算的主要原因"）
- 当数据被逐层追踪展开后，用户再次要求"对比预算数"时，**每一层展开的数据都应同步展示其预实对比**，无需用户重复要求

**进度反馈：**
- step1 查询 → 参考场景02的进度反馈
- 数据返回后 → 向用户汇报"正在计算预实差异"
- 输出表格 → 附带差异分析文字总结

### 会话上下文复用规则

为减少重复调用，以下信息在会话期间应缓存复用：
- `sessionId`：整个会话期间不变
- `modelNumber`：用户确认后整个会话不变
- 已获取的追踪数据树：后续追踪应先检查树中是否已有结果

## Error Handling

| 场景 | 处理策略 |
|------|---------|
| `bgm_ai_globalId_create` 调用失败 | 重试一次；再次失败则提示"获取会话id失败，请稍后重试" |
| `bgm_ai_model_query` 返回空列表 | 提示"当前账号暂无 EPM 数据权限，请联系管理员开通" |
| `bgm_ai_member_match` 返回空对象 | 跳过 dimAndMemberList 组装，继续后续流程 |
| `bgm_ai_member_match` 超时（Read timed out） | 若已有其他关键词匹配成功，直接使用已有结果；若全部超时则跳过 member_match，使用 template_query 降级查询 |
| `bgm_ai_template_query` 返回空列表 | 提示"未找到与问题相关的模板，请尝试换个说法或检查权限" |
| `bgm_ai_template_structure_query` 返回维度成员为空 | 提示"模板配置异常，请联系管理员检查模板维度设置" |
| `bgm_ai_data_trace` 返回空数据 | 提示"未查询到符合条件的数据，请检查维度成员范围" |
| `bgm_ai_data_node_trace` 传入 INPUT 类型单元格 id | 提前拦截，提示"录入数据无法继续追踪，已到达最底层" |
| 用户选择的维度成员组合不合法 | 提示具体错误原因，引导用户重新选择 |
| 网络超时或 MCP 工具无响应 | 提示"数据查询超时，请稍后重试"，记录超时上下文供排查 |
| 用户查询跨度超过 24 个月 | 提示"查询期间跨度较大，建议按年度或半年度汇总查询以获得更好的性能和可读性"，引导用户缩小范围或切换汇总粒度 |
| 递归追踪超过 8 层 | 停止递归，尝试利用现有数据回复用户问题，并提供追踪建议 |
| 单次 items 数组包含超过 12 个查询对象 | 拆分为多次调用（每次不超过 12 个），依次执行后合并结果 |

## Output Format

当获取到的单元格数据不为空时，按以下格式输出：

### 格式一：利用表格展示同类数据的差异对比

第一行展示多个单元格的共同维度，表格展示差异维度和数值。

**字段说明：**

| 字段 | 类型 | 说明 | 数据来源 |
|------|------|------|---------|
| 共同维度行 | string | 展示多个单元格共有的维度信息（组织/数据类型/版本） | `description` 字段解析 |
| 差异维度列（如期间） | string | 区分不同单元格的维度（期间/科目/组织等） | `description` 字段解析 |
| 数值列（如营业收入） | number | 对应维度下的单元格值，按数字格式规则展示 | `value` 字段 |

**示例：**

组织：环宇家电广州分公司、数据类型：实际数、版本：实际数版本

|   期间   | 营业收入        | 营业总成本        | 营业利润        |
| :----------- | :----------- | :-------------- | :-------------- |
| 2026 年 1 月 | 4,455,500.00 | 3,178,000.00 | 1,277,500.00 |
| 2026 年 2 月 | 2,058,441.00 | 2,000,000.00 | 58,441.00 |

### 数字格式规则

- 金额类数值使用千分位分隔符，保留 2 位小数（如 4,575,500.00）
- 数量、单价等整数字段不显示小数位（如 1,100）
- 百分比保留 2 位小数并添加 % 后缀（如 95.00%）

### 格式二：依赖链路图

当涉及多层级追溯时，使用树形链路图展示完整的数据依赖关系。每个节点后标注其类型：`【聚合】`、`【公式】` 或 `【录入】`。对于 `aggType:2` 的相减节点，可在树形图中用"减："前缀明确其减项性质。

**节点字段说明：**

| 字段 | 类型 | 说明 | 数据来源 |
|------|------|------|---------|
| 节点名称 | string | 维度成员名称（如"营业收入"） | `description` 字段解析 |
| 节点数值 | number | 该单元格的值，括号展示 | `value` 字段 |
| 节点类型标签 | enum | `【聚合】`/`【公式】`/`【录入】` | `cellType` 字段（AGG/RULE/INPUT） |
| 减项前缀 | string | 当 `aggType:2` 时展示"减："前缀 | `aggType` 字段 |
| 公式说明 | string | 公式类型节点展示计算逻辑 | `rule.ruleShow` 字段 |

**公式节点 `ruleShow` 解析规则：**
- `ruleShow` 格式为 `${id1}+${id2}` 或 `${id1}*0.95` 等，其中 `${id}` 引用该节点 `rule.children` 中对应 `id` 的单元格
- 解析方法：
  1. 从 `rule.children` 中按 `id` 找到对应的子节点
  2. 将子节点的名称（从 `description` 解析）替换 `${id}` 引用
  3. 若无法获取子节点名称，保留公式原文并在注释中说明
- 示例：`ruleShow: "=${984052128200588672}*0.95"`，且 `rule.children[0].description` 为"签约金额"
  → 展示为：**主营业务收入【公式 = 签约金额 × 0.95】**

**示例：**

```properties
三、营业利润（亏损以"-"号填列） (1,747,500.00) 【聚合】
│
├── 一、营业收入 (4,455,500.00) 【聚合】
│   │
│   └── 其中：主营业务收入 (4,455,500.00) 【公式 = 签约金额 × 0.95】
│       │
│       └── 签约金额 (4,690,000.00) 【聚合】
│
└── 减：营业总成本 (2,708,000.00) 【聚合】
    │
    └── 主营业务成本 (2,708,000.00) 【公式 = 结转成本】
        │
        └── 结转成本 (2,708,000.00) 【聚合 业态合计】
            │
            └── 住宅 (2,708,000.00) 【聚合】
                │
                ├── 超高层住宅 (1,306,000.00) 【规则】
                ├── 中高层住宅 (282,000.00) 【规则】
                └── 高层住宅 (1,120,000.00) 【规则】
```

### 格式三：组织结构追溯图

当从组织维度分析数据来源时，展示从集团到最底层组织的利润贡献链路，每个节点标注贡献占比。

**节点字段说明：**

| 字段 | 类型 | 说明 | 数据来源 |
|------|------|------|---------|
| 组织名称 | string | 组织维度成员名称 | `description` 中 Entity 维度成员名 |
| 组织数值 | number | 该组织节点的值 | `value` 字段 |
| 贡献占比 | percentage | 该组织占父节点的百分比 | 计算值：子节点 value / 父节点 value × 100% |
| 节点类型标签 | enum | `【聚合】`/`【无数据】` | `cellType` 字段，无数据时标注 `【无数据】` |
| 子科目明细 | number | 关键科目的值（如营业收入/成本） | 子节点 `value` 字段 |

**示例：**

```properties
环宇集团 (1,277,500.00) 【聚合】
│
└── 环宇家电 (1,277,500.00 / 100.00%) 【聚合】
    │
    ├── 环宇家电广州分公司 (1,277,500.00 / 100.00%) 【聚合】
    │   ├── 营业收入: 4,455,500.00
    │   └── 营业总成本: 3,178,000.00
    │
    └── 环宇家电深圳分公司 (0.00 / 0.00%) 【无数据】
```

### 输出选择策略

- 用户提问只涉及查询时，优先返回格式一且只返回查询数据不返回上游数据
- 用户提问涉及追踪、分析时，根据追踪方向选择格式二或格式三
- 用户提问涉及组织结构/利润来源/贡献分析时，优先返回格式三
- 分析类输出应附带简要的文字总结（如趋势判断、异常提示、主要贡献方等）

用户输入："广州公司 1-4 月营业利润"

期望行为：

1. 参考示例 1 获取数据的实际数

**示例 4：追踪过程中按产品业态展开明细**

2. 使用已匹配的科目编码 YYLR 调用 template_query
3. 从 template_structure_query 返回的组织维度成员中，通过名称匹配找到"广州分公司"(HYJD_GZ)
4. 组装完整维度调用 data_trace 查询数据

**示例 6：连续预实对比（查询→对比→追踪→再对比）**

- "广州公司 1 月营业利润" → 查询实际数

- "追踪一下结转成本的构成" → 按产品业态维度展开，每一层同时展示预实对比

## Rules

- **禁止编造数据**：所有输出必须基于 `bgm_ai_data_trace` 或 `bgm_ai_data_node_trace` 的真实返回结果，严禁虚构数值、维度成员或模板信息。数据缺失时必须标注"数据暂缺"，不得用估算值替代
- **权限边界**：本 Skill 仅执行数据查询和展示操作，不修改、删除或写入任何 EPM/ERP 系统中的数据
- **输入校验**：用户输入的期间编码必须符合 BudgetPeriod 编码规则（FY[年份].[时间段编码]），不符合时应引导用户修正而非猜测。详细规则见 `reference/tool-reference.md` 的"预算期间维度编码规则"
- **数据脱敏**：若查询结果涉及敏感财务数据，在对外分享或生成报告时，应提醒用户确认数据分享范围
- **缓存复用**：同一会话内，sessionId 和 modelNumber 确认后不得重复创建；已获取的追踪数据树应优先复用，避免重复调用工具
- **渐进式披露**：查询结果较大时，优先展示用户直接提问相关的数据，上游追踪数据仅在用户明确要求或分析场景下展开
- **语言一致**：与用户交互时保持中文；涉及专业术语（如 cellType、aggType）时可保留英文并附中文注释
- **幂等性**：对同一查询条件的重复执行应产生一致结果，不应因调用次数不同而改变输出格式或内容
